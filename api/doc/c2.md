# C2 : Recursos e Base de Dados

A API disponibiliza 6 recursos principais na base de dados: Utilizadores, Planos de Treino, Exercícios, Sessões, Avaliações Físicas e Metas. 

Cada recurso implementa as operações *CRUD* (*Create*, *Read*, *Update*, *Delete*) através dos métodos *HTTP* *POST*, *GET*, *PUT* e *DELETE*, seguindo os princípios de uma arquitetura *REST*. 

A API foi desenvolvida com *Node.js* e *Express 4*, utilizando o ORM *Sequelize 6* para interação com a base de dados *MySQL 8.0*, e inclui autenticação via *Passport.js* com suporte a *OAuth2* (*GitHub* e *Google*), *API Key* e *Basic Auth*.

## Tabelas da Base de Dados

### Utilizadores

Estrutura do Recurso:

```json
{
  "id": 1,
  "provider_id": "basic_1234567890_abc123",
  "provider": "basic",
  "username": "joao_fitness",
  "display_name": "João Ferreira",
  "email": "joao@gym.pt",
  "avatar_url": null,
  "api_key": "a3f9c2d1e8b7...",
  "role": "treinador",
  "createdAt": "2026-01-01T10:00:00.000Z"
}
```

Operações CRUD:

- `GET /users` - Lista utilizadores 
- `GET /users/me` - Obtém o perfil do utilizador autenticado
- `POST /auth/register` - Regista um novo utilizador com *Basic Auth*
- `POST /auth/login` - Autentica com username/email + password
- `POST /users/me/api-key` - Regenera a *API key* do utilizador autenticado
- `PUT /users/:id/role` - Altera o role de um utilizador *(admin)*
- `GET /auth/github` - Inicia autenticação *OAuth2* via *GitHub*
- `GET /auth/google` - Inicia autenticação *OAuth2* via *Google*
- `GET /auth/logout` - Termina a sessão

Filtros de Query:

- `GET /users?cliente_id=7` - Filtra (quando aplicável) pelo `id` de cliente.
- `GET /users?treinador_id=2` - **Apenas admin**: filtra clientes por treinador.

Campos da tabela:   

| Campo | Tipo | Obrigatório | Descrição |
|---|---|---|---|
| `id` | INT | Auto | Identificador único do utilizador |
| `provider_id` | VARCHAR UNIQUE| Sim | ID único do utilizador no provider |
| `provider` | ENUM | Sim | Origem da conta: `github`, `google`, `basic` |
| `username` | VARCHAR | Sim | Username único |
| `display_name` | VARCHAR | Não | Nome completo |
| `email` | VARCHAR | Não | Endereço de email |
| `password_hash` | VARCHAR | Não | Hash `bcrypt` da *password*  |
| `avatar_url` | VARCHAR | Não | URL da foto de perfil |
| `api_key` | VARCHAR UNIQUE | Não  | Chave para autenticação via header `X-API-Key`  |
| `role` | ENUM | Sim | Papel do utilizador: `admin`, `treinador`, `cliente` |
| `created_at` | TIMESTAMP | Auto | Data de criação do registo |
| `updated_at` | TIMESTAMP | Auto | Data da última atualização |

### Planos de Treino

Estrutura do Recurso:

```json
{
  "id": 1,
  "titulo": "Plano de Hipertrofia - Nível Intermédio",
  "descricao": "Plano focado no aumento de massa muscular com 4 dias de treino semanais",
  "objetivo": "hipertrofia",
  "duracaoSem": 12,
  "tipo": "profissional",
  "treinadorId": 2,
  "clienteId": 5,
  "ativo": true,
  "createdAt": "2026-01-01T10:00:00.000Z"
}
```

Operações CRUD:

- `GET /planos` - Lista planos 
- `GET /planos/:id` - Obtém um plano específico pelo seu ID
- `POST /planos` - Cria um novo plano de treino
- `PUT /planos/:id` - Atualiza um plano de treino
- `DELETE /planos/:id` - Remove um plano 

Filtros de Query:

- `GET /planos?cliente_id=7` - Filtra planos por cliente.
- Se o utilizador autenticado for **cliente**, o filtro `cliente_id` não é permitido e a listagem é sempre apenas dos seus planos.
- Se o utilizador autenticado for **treinador**, sem filtros só vê os planos em que é responsável; com `cliente_id` só pode ver planos profissionais do seu âmbito (e clientes sem qualquer plano profissional associado a treinador).

Campos da Tabela:

| Campo | Tipo | Obrigatório | Descrição |
|---|---|---|---|
| `id` | INT PK | Auto | Identificador único do plano |
| `titulo` | VARCHAR | Sim | Título do plano |
| `descricao` | TEXT | Não | Descrição do plano |
| `objetivo` | ENUM | Sim | Objetivo: `emagrecimento`, `hipertrofia`, `resistencia`, `flexibilidade`, `saude_geral` |
| `duracao_sem` | INT | Sim | Duração em semanas |
| `tipo` | ENUM | Sim | Tipo do plano: `pessoal`, `profissional` (padrão: `profissional`) |
| `treinador_id` | INT FK | Não | Referência ao treinador responsável (relação N:1) |
| `cliente_id` | INT FK | Sim | Referência ao cliente destinatário (relação N:1) |
| `ativo` | BOOLEAN | Sim | Indica se o plano está ativo (padrão: `true`) |
| `created_at` | TIMESTAMP | Auto | Data de criação do registo |
| `updated_at` | TIMESTAMP | Auto | Data da última atualização |

### Exercícios

Estrutura do Recurso:

```json
{
  "id": 1,
  "planoId": 1,
  "nome": "Supino Reto com Barra",
  "grupoMuscular": "peito",
  "series": 4,
  "reps": 10,
  "pesoKg": 80.00,
  "notas": "Manter as escápulas retraídas durante o movimento",
  "ordem": 1
}
```

Operações CRUD:

- `GET /exercicios` - Lista exercícios de um plano 
- `GET /exercicios/:id` - Obtém um exercício 
- `POST /exercicios` - Cria um novo exercício
- `PUT /exercicios/:id` - Atualiza um exercício 
- `DELETE /exercicios/:id` - Remove um exercício 

Filtros de Query:

- `GET /exercicios?plano_id=1` - Lista todos os exercícios de um plano.
- Se o utilizador autenticado for **cliente**, o `plano_id` pode ser omitido; a API escolhe automaticamente um dos seus planos.
- **Admin/Treinador:** pode usar `cliente_id` para indicar o cliente (e, se necessário, a API escolhe um plano desse cliente).
- **Treinador:** ao usar `cliente_id`, só funciona para clientes do seu âmbito e para clientes sem qualquer plano profissional associado a treinador.
- Se o cliente não tiver planos, a API devolve uma mensagem a indicar que não existem exercícios associados.


Campos da Tabela:

| Campo | Tipo | Obrigatório | Descrição |
|---|---|---|---|
| `id` | INT PK | Auto | Identificador único do exercício |
| `plano_id` | INT FK | Sim | Referência ao plano de treino - relação 1:N principal |
| `nome` | VARCHAR | Sim | Nome do exercício |
| `grupo_muscular` | VARCHAR | Sim | Grupo muscular trabalhado |
| `series` | INT | Sim | Número de séries |
| `reps` | INT | Sim | Repetições por série |
| `peso_kg` | DECIMAL | Não | Carga em quilogramas |
| `notas` | TEXT | Não | Notas ou instruções adicionais |
| `ordem` | INT | Sim | Ordem do exercício no plano (padrão: `1`) |

### Sessões

Estrutura do Recurso:

```json
{
  "id": 1,
  "planoId": 1,
  "clienteId": 5,
  "treinadorId": 2,
  "dataSessao": "2026-04-15",
  "duracaoMin": 60,
  "notas": "Cliente apresentou boa evolução no supino",
  "estado": "concluida"
}
```

Operações CRUD:

- `GET /sessoes` - Lista sessões 
- `GET /sessoes/:id` - Obtém uma sessão específica pelo seu ID
- `POST /sessoes` - Regista uma nova sessão 
- `PUT /sessoes/:id` - Atualiza uma sessão
- `DELETE /sessoes/:id` - Remove uma sessão 

Filtros de Query:

- `GET /sessoes?estado=concluida` - Filtra por estado (`agendada`, `concluida`, `cancelada`).
- `GET /sessoes?plano_id=1` - Filtra por plano.
- `GET /sessoes?cliente_id=7` - Filtra por cliente (**apenas admin/treinador**).

Campos da Tabela:

| Campo | Tipo | Obrigatório | Descrição |
|---|---|---|---|
| `id` | INT PK | Auto | Identificador único da sessão |
| `plano_id` | INT FK | Sim | Referência ao plano de treino (relação N:1) |
| `cliente_id` | INT FK | Sim | Referência ao cliente (relação N:1) |
| `treinador_id` | INT FK | Não | Referência ao treinador (relação N:1, opcional) |
| `data_sessao` | DATE | Sim | Data em que a sessão foi realizada |
| `duracao_min` | INT | Sim | Duração da sessão em minutos |
| `notas` | TEXT | Não | Notas do treinador sobre a sessão |
| `estado` | ENUM | Sim | Estado: `agendada`, `concluida`, `cancelada` (padrão: `agendada`) |

## Relações entre os Recursos

### Relações 1:N (Um para Muitos)

Todas as relações no projeto são do tipo 1:N, implementadas através de chaves estrangeiras:

- **Utilizadores → Planos de Treino**: Um treinador cria muitos planos; um cliente tem muitos planos atribuídos
  - Implementada através dos campos `treinador_id` e `cliente_id` na tabela `planos_treino`

- **Planos de Treino → Exercícios**: Um plano tem muitos exercícios
  - Implementada através do campo `plano_id` na tabela `exercicios`
  - Configurada com `CASCADE DELETE`: ao apagar um plano, os seus exercícios são removidos automaticamente

- **Planos de Treino → Sessões**: Um plano tem muitas sessões registadas
  - Implementada através do campo `plano_id` na tabela `sessoes`

- **Utilizadores → Sessões**: Um cliente realiza muitas sessões; um treinador supervisiona muitas sessões
  - Implementada através dos campos `cliente_id` e `treinador_id` na tabela `sessoes`

- **Utilizadores → Avaliações Físicas**: Um cliente tem muitas avaliações físicas
  - Implementada através do campo `cliente_id` na tabela `avaliacoes_fisicas`

- **Utilizadores → Metas**: Um cliente tem muitas metas definidas
  - Implementada através do campo `cliente_id` na tabela `metas`

- **Planos de Treino → Metas**: Um plano pode ter muitas metas associadas
  - Implementada através do campo `plano_id` na tabela `metas`
