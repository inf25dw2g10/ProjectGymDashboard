# C3 : Autenticação e Integração com a API (Frontend)

## Introdução

O frontend não implementa autenticação no servidor - **consome** os mecanismos expostos pela GymAPI e gere a sessão no browser:

- **Basic Auth** via `POST /auth/login` → recebe `apiKey` + `role`
- **OAuth 2.0** via redirect para `/auth/github` ou `/auth/google` → callback com credenciais
- **API Key** enviada em todos os pedidos no header `X-API-Key`

A **autorização por role** é aplicada no frontend através de `RoleRoute` e na API através dos controllers.

## Fluxo OAuth 2.0 (frontend)

1. Em `/login`, o utilizador clica em GitHub ou Google - `src/pages/LoginPage.jsx` linhas 128-145
2. O browser segue o link `/auth/github` ou `/auth/google` (proxy → API)
3. A API redireciona para o provider e, após login, volta ao callback da API
4. A API redireciona para `/oauth/callback?apiKey=...&role=...&userId=...`
5. `OAuthCallbackPage` guarda credenciais e redireciona - `src/pages/OAuthCallbackPage.jsx` linhas 19-53

apiClient.interceptors.request.use(
  (config) => {
    const apiKey = AuthService.getApiKey();
    if (apiKey) {
      config.headers['X-API-Key'] = apiKey;
    }
    return config;
  },
  (error) => Promise.reject(error)
);

apiClient.interceptors.response.use(
  (response) => response,
  (error) => {
    if (error.response?.status === 401) {

## Rotas protegidas por role

### `RoleRoute`

Verifica autenticação **e** role antes de renderizar o dashboard - `src/routes/RoleRoute.jsx` linhas 23-39:

```23:39:src/routes/RoleRoute.jsx
const RoleRoute = ({ component: Component, allowedRoles, ...rest }) => (
  <Route
    {...rest}
    render={(props) => {
      if (!AuthService.isLoggedIn()) {
        return <Redirect to="/login" />;
      }

      const role = AuthService.getUserRole();

      if (!allowedRoles.includes(role)) {
        return <Redirect to="/unauthorized" />;
      }

      return <Component {...props} />;
    }}
  />
);
```

## Autorização no frontend vs API

| Camada | Responsabilidade |
|--------|------------------|
| **Frontend** (`RoleRoute`) | Impede acesso ao dashboard de outra role |
| **API** (controllers) | Garante que cada utilizador só vê/edita os seus recursos |

O frontend **não substitui** a autorização da API - apenas melhora a experiência do utilizador. Mesmo que alguém aceda manualmente a um endpoint, a API devolve `403` se a role não tiver permissão.
